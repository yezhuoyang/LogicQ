# MagicQ Design Plan

MagicQ is the protocol language for producing, improving, and consuming magic
states while their QEC carrier can change over time. It should sit beside the
current logical compiler: algorithm compilation may emit typed `MagicObligation`s,
while MagicQ compiles factory/cultivation protocols that discharge those
obligations.

## Protocol Facts To Encode

Cultivation is a live-carrier protocol, not just a resource transformer.
Gidney-Shutty-Jones split it into injection, cultivation, and escape:

- Injection creates an encoded `T` state in a distance-3 color code. The paper
  compares degenerate/teleport, Bell-growth, and unitary injection; the unitary
  injection is the default source-code path.
- Cultivation uses a check-grow-stabilize cycle. The check is a double-check of
  the logical value, growth enlarges the color code using Bell-pair initialized
  new data qubits, and stabilization repeats superdense color-code cycles.
- Early stages use full postselection: any detector event discards the attempt.
  Fast feedback remains needed to force newly prepared stabilizers into the
  required eigenstate before the next layer of `T` gates.
- Escape is a code-growth/code-switching protocol. The cultivated state is too
  reliable for the small color code, so the protocol grafts it into a larger
  surface/matchable carrier, measures rounds, postselects selected color-region
  detectors, then keeps/rejects based on a decoder gap.
- The paper uses `d1` for target fault distance at the end of cultivation,
  `d2` for the final code distance after escape, `r1` for grafted-code rounds,
  and `r2` for final matchable-code rounds.

The canonical 15-to-1 scheme comes from the Bravyi-Kitaev 15-qubit punctured
Reed-Muller CSS code:

- It consumes 15 noisy `T`-basis magic states and, on success, produces 1 better
  `T` state.
- The code is described by punctured Boolean-function tables. `L1` is generated
  by the four linear functions `[x_j]`; `L2` is generated by those plus the six
  quadratic functions `[x_i x_j]`.
- The abstract protocol measures 10 `Z`-type checks from `[x_j]` and
  `[x_i x_j]`, computes a correction `A(w)^dagger`, measures 4 `X/A` checks,
  postselects on trivial `eta`, then decodes one logical qubit.
- Success probability and output error can be carried symbolically:
  `p_s = (1 + 15 * (1 - 2 eps)^8) / 16`,
  and for small `eps`, `eps_out ~= 35 * eps^3`; the threshold is about `0.141`.
- MagicQ should name this as `RM15To1` or `ReedMuller15To1` and track the
  convention that Bravyi-Kitaev's `A0` state is the usual `T|+>` state up to
  global phase/Clifford naming.

## Language Shape

MagicQ v1 should have two protocol families:

```text
protocol cultivate_T(d1, d2, r1, r2, basis := T):
  inject unitary into ColorCode(3)
  check logical_value using double_cat
  grow ColorCode(3) -> ColorCode(d1) by Bell-boundary growth
  stabilize superdense cycles := 3
  escape by graft ColorCode(d1) into Matchable(d2)
  postselect early := all_detectors
  postselect escape := selected_color_detectors and gap >= threshold
  return MagicState(T, carrier := Matchable(d2), faultDistance >= d1)

protocol rm15_to_1(inputs : 15 * MagicState(T)):
  use ReedMuller15 code
  measure syndrome
  postselect eta = 0
  decode
  return MagicState(T, error := 35 * eps^3 + higher_terms)
```

This first syntax can be represented directly in Lean as records and inductive
operations before committing to a parser.

## Core Types

Add a `MagicQ` Lean namespace with these data layers:

- `MagicBasis`: `T`, `Tdg`, `Y`, `A0`, and Clifford-equivalent aliases.
- `MagicQuality`: claimed raw/input error, output error bound, success
  probability, code distance, and fault distance.
- `Carrier`: a live QEC carrier containing a `TypeChecker.Block` or a
  ChainQ-family reference, logical basis, ownership/liveness, decoder tag, and
  current `MagicQuality`.
- `MagicResource`: a linear resource `MagicState basis carrier quality` or a
  pending `MagicObligation` target from `Compiler.Mixed.Syntax`.
- `PostselectPolicy`: full detector postselection, tagged detector
  postselection, syndrome predicate, and decoder-gap predicate.
- `ProtocolOp`: `inject`, `checkLogical`, `grow`, `stabilize`, `graft`,
  `ablate`, `distill15To1`, `decode`, `postselect`, `discard`, `output`.
- `ProtocolCert`: the checkable and deferred evidence attached to each dynamic
  carrier transition.

The key judgement should thread the ordinary typed environment and a magic
resource environment:

```text
Gamma; Sigma; Post |- op => Gamma'; Sigma'; obligations
```

`Gamma` is the existing `TypedEnv`, `Sigma` is MagicQ's linear magic-resource
context, and `obligations` records deferred physical/decoder/fault-distance
claims.

## Dynamic QEC Type Rule

The essential new type former is a carrier transition:

```text
Carrier(code = C, logical = l, quality = q)
  -- grow/switch/graft -->
Carrier(code = D, logical = f(l), quality = q')
```

This should reuse the existing ChainQ and switch infrastructure:

- ChainQ provides the code family and materialized stabilizer/logical data.
- `TypeChecker.Block` provides the typed block shape and logical basis.
- `TypeChecker.Judgment.Switch` provides the algebraic cross-code check.
- `Compiler.CodeSwitch.Basic` provides richer external certificates and
  deferred obligations.

MagicQ extends those with protocol-specific obligations:

- which new stabilizers are prepared versus measured,
- which detectors are postselected,
- which logical observable is being preserved,
- what fault-distance claim is being made,
- what decoder/gap predicate is used after escape,
- whether the failure branch is disposable and can be retried.

For v1, the checker should not try to verify physical stochastic correctness.
It should reject shape/logical mismatches, ownership violations, non-linear use
of states, invalid postselection scopes, and protocol schedules that return a
state without the required carrier/quality claims.

## Cultivation V1 Plan

1. Add a ChainQ color-code placeholder implementation sufficient for distances
   3 and 5, because `ChainQ/ColorCode` is currently only a README. Start with
   explicit matrices/checked examples, then generalize the family.
2. Add `MagicQ/Syntax.lean` records for `CultivationSpec`, `CultivationStage`,
   `PostselectPolicy`, `MagicQuality`, and `Carrier`.
3. Add `MagicQ/Check.lean` with a checker that threads carrier state through:
   `Inject -> Check -> Grow -> Stabilize -> Escape -> Output`.
4. Model `d1`, `d2`, `r1`, `r2`, `basis`, `injectStyle`, and `decoderPolicy`
   as fields. Use defaults from the paper/source: unitary injection, full early
   postselection, superdense stabilization cycles, graft escape to matchable
   code.
5. Represent color-code growth as a `GrowCert` preserving the chosen logical
   observable and adding a fault-distance claim. For v1, accept the paper/source
   construction as an external certificate after checking shapes and logical
   arity.
6. Represent escape as a sequence of `graft`, `rounds`, `ablate`, and
   `gapPostselect`. The type result should be a matchable/surface-like carrier
   of distance `d2`.
7. Add a library declaration `MagicQ.Library.Cultivation.defaultT` that
   constructs the standard cultivation protocol and returns one `MagicState T`.

## 15-To-1 V1 Plan

1. Add a `MagicQ.Library.ReedMuller15` module with the `[x_j]` and
   `[x_i x_j]` binary vectors used in the Bravyi-Kitaev paper.
2. Build a checked `[[15,1,3]]` CSS/stabilizer block from those vectors. The
   immediate target is shape/CSS/logical-basis validity, not a full proof of the
   distillation threshold.
3. Define `RM15To1Spec` as a resource transformer:
   consumes exactly 15 compatible `MagicState T` inputs, has a failure branch
   that discards all scratch/input state, and produces exactly 1 `MagicState T`
   on success.
4. Attach symbolic quality equations: success probability, threshold
   precondition, and cubic output-error leading term.
5. Provide two lowerings:
   `canonicalSyndrome`, mirroring the paper's syndrome/check/decode sequence,
   and `gidneyPauliSchedule`, mirroring
   `testdata/factory_scripts/o1_d3_t15_q5_reedmuller.dat`.
6. The first executable lowering should target PPM-style operations where
   possible and leave any non-Clifford state input as `MagicResource`
   consumption, not as a new raw `T` obligation.

## Compiler Integration

MagicQ should integrate with the current compiler in three steps:

1. Current logical compilation emits `.magic { kind := .tGate, target := q }`.
   Preserve this path.
2. Add a `MagicSupplyPlan` that maps each `MagicObligation` to a MagicQ protocol
   output: either `cultivate_T` or `rm15_to_1`.
3. Add a discharge pass that replaces a `magic` obligation with:
   factory scheduling metadata now, and later concrete LS/QStab operations for
   state production plus gate teleportation.

This keeps MagicQ parallel to the rest of compilation. The algorithm compiler
does not need to know whether a `T` state came from cultivation or RM15 until a
resource/scheduling pass binds obligations to factory outputs.

3. Add ChainQ color-code stubs for distance 3 and 5.

7. Lower RM15 canonical syndrome to existing PPM/Mixed fragments where possible.
8. Add optional import/conversion from Gidney `FactoryScript` and cultivation

  paired with either `cultivate_T` or `rm15_to_1`.

## Known Risks

- `ChainQ/ColorCode` is not implemented yet, so cultivation cannot be made
  honestly ChainQ-typed without adding at least small explicit color-code
  constructors.
- Cultivation's escape stage relies on decoder behavior and gap predicates.
  V1 should track these as explicit external obligations instead of pretending
  they are proved.
- The cultivation paper includes an errata discussion around stabilizer signs
  and `H_XY`/basis conventions. MagicQ should therefore track basis/sign
  conventions explicitly rather than treating all `T`-like states as identical.
- RM15 has several equivalent presentations. MagicQ should support the canonical
  code/syndrome definition first and treat the compact Gidney factory script as
  a lowering fixture.
